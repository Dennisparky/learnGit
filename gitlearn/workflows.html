Git Workflows

Common branching and release strategies used in real-world teams.

What is a Workflow?

A Git workflow defines how a team uses branches, merges, and releases to build software collaboratively.

Feature Branch Workflow

In this workflow, developers create a new branch for each feature or fix, then merge back into the main branch when the work is complete.

Real-world scenario

Imagine a team building a new login experience. Each developer creates a branch like feature/login-ui or feature/oauth, works independently, and opens a pull request when ready.

Key steps

  • git checkout -b feature/your-feature — Create and switch to a feature branch.
  • Work on the feature and commit changes.
  • git checkout main and git merge feature/your-feature — Merge into the main branch.

GitHub Flow

GitHub Flow is a lightweight workflow centered around short-lived branches and pull requests.

Real-world scenario

A developer fixes a bug and creates a branch named fix/auth-bug. They push the branch to GitHub, open a pull request, and teammates review before merging.

Key steps

  • git checkout -b fix/auth-bug
  • Commit changes and git push to the remote.
  • Open a pull request, review, then merge into main.

Git Flow

Git Flow provides a structured approach with long-lived branches for development, releases, and hotfixes.

Real-world scenario

A project has multiple releases. Developers work on a develop branch, create feature/* branches, and prepare releases on a release/* branch.

Key branches

  • main — Production-ready code.
  • develop — Latest development changes.
  • feature/* — New features.
  • release/* — Preparing a new release.
  • hotfix/* — Quick patches to production.
